# Survey Roster Matcher App Spec

## Goal

Build a local desktop app for reviewing and confirming links between deduplicated Qualtrics survey responses and canonical roster pupils. The app must prioritise speed, auditability, and prevention of accidental many-to-one matches.

## Recommended Stack

- Desktop shell: Electron
- Frontend: React, Vite, TypeScript
- UI: Tailwind CSS and shadcn/ui
- Database: SQLite
- SQLite access: `better-sqlite3` from the Electron main process
- CSV import/export: Node CSV parser/writer
- Packaging: `electron-builder`
- Validation: Zod

The app should avoid a separate backend server unless needed later. Electron main process should own filesystem/database access; the React renderer should handle UI.

## Aesthetic And UX Direction

The app should feel slick, polished, professional, and easy to use. The design should be quiet and work-focused rather than decorative: dense but readable information, clear hierarchy, strong keyboard ergonomics, restrained colour, and minimal visual noise.

Use a modern operational-tool aesthetic: crisp typography, compact controls, clear spacing, fast scanning, and obvious state changes. Candidate matches should be easy to compare at a glance. Warnings and conflicts should be visually prominent without overwhelming the main workflow.

The interface should support both careful review and rapid happy-path processing. Obvious matches should be acceptably handled with repeated `Enter` presses, while uncertain cases should clearly expose evidence, alternatives, and conflict warnings.

## Primary Workflow: Response Review

The main screen serves one unmatched survey response at a time.

For each response, show:

- entered forename
- entered school
- entered birth month/year
- normalized values
- completion percentage
- response ID
- duplicate-attempt summary
- expandable dedupe audit

Show ranked roster candidates with:

- pupil forename and surname
- school
- DOB
- confidence score
- candidate rank
- reason chips, such as `school exact`, `DOB month/year match`, `forename exact`, `fuzzy school`, `nickname`
- warnings, such as `already matched`, `DOB mismatch`, `missing DOB`

If one high-confidence candidate exists and there is no conflict, preselect it.

## Primary Actions

- accept selected match
- defer response
- mark no roster match
- mark ambiguous
- reject selected candidate
- manual roster search
- undo last decision

Accepted match behaviour:

- links response to selected roster child
- removes response from unmatched queue
- removes roster child from normal available pool
- writes decision and audit event
- does not delete source data

Matched roster children may still appear in manual search, but with a clear warning.

## Keyboard Shortcuts

- `Enter`: accept selected match
- `Up` / `Down`: move candidate selection
- `/`: focus roster search
- `D`: defer
- `N`: no match
- `A`: ambiguous
- `R`: reject selected candidate
- `U`: undo last decision
- `Esc`: clear search / close overlay

## Queues

- unreviewed
- deferred
- ambiguous
- no-candidate
- conflict
- completed

Once the unreviewed queue is empty, the app cycles back to deferred responses.

## Secondary Views

### Roster Coverage

- all roster pupils
- matched/unmatched status
- linked response if matched

### Conflict Queue

- multiple responses claiming the same pupil
- matched pupil appearing as a strong candidate elsewhere

### School Dashboard

- roster count
- matched count
- unmatched roster pupils
- unmatched responses
- ambiguous responses

### Alias Management

- school aliases
- name aliases
- nickname mappings
- ability to rerun candidate generation after alias edits

### Audit And Export

- final match table
- roster-with-response table
- unmatched roster pupils
- unmatched survey responses
- ambiguous decisions
- rejected candidates
- full audit log

## Preselection Rule

A candidate may be preselected only if:

- confidence exceeds the high threshold
- top score is clearly ahead of the second score
- roster child is not already matched
- no hard contradiction exists
- candidate was generated by a strong gate, not weak fuzzy similarity alone

## Persistence

Use a local SQLite database stored beside the project/app data.

Every decision records:

- response ID
- roster child ID, if applicable
- decision type
- selected candidate rank
- confidence score
- reviewer action
- timestamp
- previous decision if revised
- optional note

## Design Principle

The app should support rapid `Enter`-driven acceptance for obvious cases, but never silently overwrite or hide conflicts. All automation suggests; the reviewer decides.
